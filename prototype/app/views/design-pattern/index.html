Deliver data-driven funding statements

A pattern for automating funding information to organisations.

When to use this pattern

Use this pattern if you need to automate the communication of funding information to organisations. For example, education providers or responsible bodies.

Use this pattern to communicate funding information direct to ‘child’ or independent education providers.

When not to use this pattern

Do not use this pattern if you are communicating funding to individuals (for example students receiving a loan or bursary directly).

Do not use this pattern to communicate complex funding information if the user will need access to calculations to understand what is being shown.

Do not use this pattern to communicate a combined view of funding for ‘child’ providers to a ‘parent’ organisation (for example, provider groups, multi-academy trusts (MAT) or local authorities.

How it works

This pattern is a data-driven approach to generating funding statements. The intention is for this pattern to be reusable and scalable across different funding types. The rules created to support this pattern should be global, meaning they will work for the widest range of funding types.

To implement this pattern effectively you need to follow this process:

  1. Identify the primary data source
  2. Map the data output to the required properties (see below)
  3. Define global rules for which data should be pulled through into the pattern
  4. Identify if the data will need to be transformed to be accessible, clear and meaningful to users. You should consider the following:
    • Can the transformation be done on the front-end?
    • Can the transformation be done within the API (if that is your technical approach)?
    • Does a change need to be made up-stream in the data source?
  5. Test the statement with users to ensure they understand the contents

Data source

This pattern has been developed using one primary data source, but it could be used to pull together and present data from multiple sources. In this case, all data sources would need to be mapped to identify the source of each property. Using multiple data sources adds to the complexity so designers are recommended to work closely with technical leads/ developers and business analysts to define the scope.

Information to show on a statement

Statement properties

  • Publish date
  • Version number
  • Latest version - whether the statement version number is current or historical

Funding properties

  • Funding stream or grant name
  • Date period - which dates the funding covers
  • Total allocation - the total amount of money allocated in the statement

Funding line items

  • Name - the name of a component of a funding allocation
  • Distribution periods - how the funding line is split across different time periods
  • Distribution period amounts - the amount of funding allocated for each time period
  • Sub-total - the total amount of money allocated in the funding line

Research on this pattern

This pattern was last used in user research in October 2025.

User needs for a financial statement

Based on research, we know users need a statement that:

  • Gives them the right level of detail, enabling them to identify how their allocation has been arrived at, where changes have occurred, to identify if there are any data discrepancies, and if so, where and what that discrepancy is
  • Provides contextual guidance and clearly signposts them to additional support if needed
  • Follows a clear and consistent structure, making it easy to compare information with related statements
  • Can be downloaded in an offline format to support their analysis, record keeping and future financial forecasting
  • Is transparent and accurate - This helps to build confidence and in turn, enables them to make informed and efficient decisions

Services using this pattern

  • Manage your education and skills funding - Adult statements pilot in March 2026
  • Manage my education estate private beta

Roadmap

We expect to add more information to the pattern in the first half of 2026, including:

  • How allocations are calculated
  • How to link a contract and which contract relates to the statement
  • When teh user should expect to be paid and how much
  • Providing a direct link to the relevant guidance

We also expect to develop variants of this pattern to:

  • meet the needs of users in parent organisations, for example local authorities, multi-academy trusts and responsible bodies
  • accommodate more complex funding information
